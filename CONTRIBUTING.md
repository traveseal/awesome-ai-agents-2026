# Contributing to Awesome AI Agents 2026

Thanks for helping improve Awesome AI Agents 2026. This repository is meant to be a useful, high-signal map of the AI agents ecosystem, not a generic product directory.

The goal is simple: help builders quickly discover credible AI agents, frameworks, platforms, tools, protocols, and resources that are useful in 2025-2026.

## What belongs here

A submission should clearly fit at least one of these groups:

- AI agents, agent frameworks, agent runtimes, orchestration tools, or multi-agent platforms.
- Developer tools that directly improve agent workflows, such as coding agents, browser agents, evaluation tools, tracing, safety, governance, or MCP tooling.
- Publicly available products, open-source projects, research-backed tools, or useful learning/community resources.
- Tools that are currently maintained or actively usable.

## What does not belong

Please do not submit:

- Abandoned projects with no meaningful updates in the last 6 months.
- Purely theoretical projects with no usable product, public demo, source code, or documentation.
- Duplicate entries, near-duplicate PRs, or minor forks without a clear difference.
- Generic AI tools that are not meaningfully related to agents.
- Thin landing pages, lead-capture pages, agencies, consulting offers, or services with no clear agent product.
- Promotional claims that cannot be verified.
- Tracking-heavy URLs, affiliate links, or UTM campaign links.

## Quality bar

Good entries are short, useful, and factual. A maintainer should be able to understand the tool in one sentence.

Prefer this:

```markdown
| [Tool Name](https://official-link.com) | Open-source MCP server that connects agents to business data sources across Stripe, Shopify, and analytics APIs. | Free / OSS |
```

Avoid this:

```markdown
| [Tool Name](https://official-link.com) | Revolutionary world-changing platform that will transform how every company uses AI forever. The fastest and best agent solution in the market. | Contact us |
```

## Formatting rules

- Follow the table format used in the target section.
- Keep descriptions concise: usually 1 sentence, 2 only when needed.
- Use the official website or official GitHub repository as the primary link.
- Use neutral pricing labels, such as `Free`, `Free / OSS`, `Free / Paid`, `Usage-based`, `Enterprise`, or `Custom`.
- Add the entry in the most relevant existing section.
- Keep the section ordering consistent. If a section is alphabetized, place the entry alphabetically.
- Do not modify unrelated entries in the same PR unless the PR is specifically for cleanup.

## Claims and evidence

Claims should be factual and conservative.

Allowed:

- License, language, supported integrations, pricing model, public repo, documentation, or official product capabilities.
- Measured claims only when sourced from the official project or a credible public benchmark.

Avoid:

- "Best", "fastest", "#1", "most advanced", "enterprise-grade", "revolutionary", or similar marketing language.
- Subscriber counts, revenue, valuation, adoption, or benchmark claims without a source.
- Hype around future features that are not publicly available.

## Disclosure

If you are affiliated with the project, say so in the PR body. Affiliation is allowed; hidden promotion is not.

Examples:

- "Disclosure: I maintain this project."
- "Disclosure: I work at the company behind this tool."
- "No affiliation. I found this useful."

## Pull request checklist

Before submitting a PR, confirm that:

- [ ] The entry is directly related to AI agents or agent workflows.
- [ ] The link points to an official source.
- [ ] The description is concise and factual.
- [ ] The pricing label matches the section style.
- [ ] There is no duplicate entry or duplicate PR for the same project.
- [ ] There are no UTM, referral, or affiliate parameters in the URL.
- [ ] The PR changes only the relevant lines.
- [ ] Any affiliation is disclosed in the PR body.

## Suggested issue-first cases

Open an issue before submitting a PR when:

- You want to propose a new category.
- The project fits multiple sections and placement is unclear.
- The submission is a commercial service rather than a product/tool.
- The entry requires a larger README reorganization.
- You are unsure whether the project is agent-specific enough.

## Review process

Maintainers may approve, request changes, or close submissions based on fit and quality. Common reasons a PR may not be approved immediately:

- Wrong section.
- Duplicate or near-duplicate submission.
- Promotional language.
- Broken or non-official link.
- Too much unrelated README churn.
- Encoding changes such as BOM characters.
- Merge conflicts or outdated branch.

## Updating existing entries

Corrections are welcome. Please submit a focused PR for:

- Broken links.
- Pricing changes.
- Project renames.
- Better descriptions.
- Moved repositories.
- Clearly outdated claims.

## Code of conduct

Be respectful and constructive. The maintainer's job is to protect the usefulness of the list. Not every product belongs here, and that is what keeps the repository valuable.
